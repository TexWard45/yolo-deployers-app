# Engineering Spec: Discord Channel Message Auto-Ingestion

## 1. Job to Be Done

- **Who**: Workspace admins who want to capture Discord channel messages as support conversations automatically.
- **What**: The system should listen to configured Discord channels via a bot, receive every non-bot message in real time, and persist it into the database as a `ConversationMessage` linked to a `Conversation` and `CustomerProfile` — without any manual intervention.
- **Why**: Currently the Discord bot (`apps/queue/src/discord-bot.ts`) already does inline ingestion directly in the `MessageCreate` handler. However, this approach has several problems:
  1. **No Temporal workflow orchestration** — the existing `ingestSupportMessageWorkflow` and its activities (`ingestSupportMessage`, `checkAutoGenerateDraft`) exist but the Discord bot bypasses them entirely, duplicating all the ingestion logic inline.
  2. **No auto-draft trigger** — because the workflow is bypassed, the auto-generate-draft feature (configured via `WorkspaceAgentConfig.autoDraftOnInbound`) never fires on Discord messages.
  3. **No retry/resilience** — if a DB write fails in the bot handler, the message is silently lost. Temporal would provide automatic retries.
  4. **Channel filtering is hardcoded** — the bot only listens to channels with "support" or "company" in their name instead of using the `ChannelConnection.configJson` to define which channels to monitor.
  5. **No per-channel connection scoping** — `externalAccountId` is matched against guildId OR channelId loosely, rather than checking a configured channel list.
- **Success criteria**:
  - Every non-bot message in a configured Discord channel is persisted as a `ConversationMessage` within seconds.
  - Messages are processed through the Temporal `ingestSupportMessageWorkflow`, enabling auto-draft generation.
  - Duplicate messages are never stored (idempotency via `externalMessageId`).
  - Channel filtering is driven by `ChannelConnection.configJson` (list of channel IDs), not hardcoded name matching.
  - The system recovers gracefully from transient DB/Temporal failures via Temporal retry policies.

## 2. Proposed Flow / Architecture

### Data Model Changes

No new Prisma models are needed. The existing schema already supports this flow:

- **`ChannelConnection`** — already exists with `type: DISCORD`, `configJson` (JSON field). We will standardize `configJson` to include `{ channelIds: string[] }` for Discord connections, specifying which channel IDs the bot should listen to.
- **`Conversation`**, **`ConversationMessage`**, **`CustomerProfile`**, **`CustomerChannelIdentity`** — all exist and are used by the current inline handler.

**Zod schema addition**: Add a `DiscordChannelConfigSchema` to validate the `configJson` shape for Discord connections.

```ts
// In @shared/types schemas
export const DiscordChannelConfigSchema = z.object({
  channelIds: z.array(z.string()).min(1, "At least one channel ID required"),
  listenToThreads: z.boolean().optional().default(true),
});
```

### API Layer

**Existing tRPC routers involved** (no new routers needed):

- `channelConnection.createDiscordConnection` — update to validate `configJson` against `DiscordChannelConfigSchema`.
- `conversation.list` / `conversation.getById` — already work for viewing ingested conversations.
- `agent.getWorkspaceConfig` — already exposes `autoDraftOnInbound` toggle.

### Frontend

No frontend changes required for the core ingestion feature. The existing conversation list UI will display ingested Discord messages. Optional future enhancement: a channel connection setup form that lets admins pick Discord channels.

### Flow Diagram

**Message Ingestion Flow (revised):**

1. Discord bot connects using `DISCORD_BOT_TOKEN` on worker startup.
2. Bot receives `MessageCreate` event from Discord gateway.
3. Bot checks: is the message from a bot? If yes, skip.
4. Bot queries `ChannelConnection` records where `type = 'DISCORD'` and `status = 'active'`.
5. For each matching connection, parse `configJson` to get `channelIds`.
6. Check if `message.channelId` (or parent channel ID for threads) is in `channelIds`. If not, skip.
7. Bot starts a Temporal workflow `ingestSupportMessageWorkflow` with the message payload (instead of doing inline DB writes).
8. Temporal workflow executes `ingestSupportMessage` activity:
   a. Idempotency check via `externalMessageId`.
   b. Upsert `CustomerChannelIdentity` + `CustomerProfile`.
   c. Find or create `Conversation`.
   d. Create `ConversationMessage`.
9. Workflow checks `autoDraftOnInbound` via `checkAutoGenerateDraft` activity.
10. If enabled, workflow calls `generateReplyDraft` activity to create an AI draft reply.
11. Workflow returns `{ conversationId, draftGenerated }`.

**Connection Cache Strategy:**

To avoid querying `ChannelConnection` on every single message, the bot should maintain an in-memory cache of active Discord connections, refreshed on a configurable interval (e.g., every 60 seconds). This keeps the hot path fast.

### Dependencies

- **Existing packages** (no new installs needed):
  - `discord.js` v14 — already in `apps/queue`
  - `@temporalio/client` — already in `apps/queue`
  - `@shared/database` — already used
  - `@shared/types` — for Zod schemas

- **Environment variables** (already defined in `@shared/env`):

  - `TEMPORAL_ADDRESS`, `TEMPORAL_NAMESPACE`, `TEMPORAL_TASK_QUEUE` — workflow routing

## 4. Testing Checklist

### Happy Path

- [ ] Send a message in a configured Discord channel — verify a `ConversationMessage` is created with correct `body`, `externalMessageId`, `direction: INBOUND`, `senderKind: CUSTOMER`
- [ ] Send a message from a new Discord user — verify `CustomerProfile` and `CustomerChannelIdentity` are created
- [ ] Send a message from a returning user — verify existing `CustomerProfile` is reused, no duplicates
- [ ] Send a message when `autoDraftOnInbound` is enabled — verify a `ReplyDraft` with status `GENERATED` is created
- [ ] Send a message in a Discord thread — verify it attaches to the correct `Conversation` via `externalThreadId`

### Validation

- [ ] Creating a Discord `ChannelConnection` with empty `channelIds` array is rejected
- [ ] Creating a Discord `ChannelConnection` with invalid `configJson` shape is rejected
- [ ] Messages with empty body are still ingested (Discord allows embeds/attachments with no text)

### Edge Cases

- [ ] Duplicate message (same `externalMessageId`) — verify idempotency, no duplicate `ConversationMessage`
- [ ] Message in a channel not in any connection's `channelIds` — verify it is ignored
- [ ] Message from a bot user — verify it is ignored
- [ ] `ChannelConnection` with `status: inactive` — verify messages in that connection's channels are ignored
- [ ] Multiple active connections for same guild but different channels — verify correct connection is matched
- [ ] Temporal is temporarily unavailable — verify the bot logs the error and does not crash (message lost but bot survives)
- [ ] Connection cache refresh — verify newly added/removed channel connections take effect within 60s

### Auth / Permissions

- [ ] Only workspace admins can create/update `ChannelConnection` records (existing `protectedProcedure` enforcement)
- [ ] Bot token is not exposed in any API response or log

### Type Safety

- [ ] `npm run type-check` passes with all changes
- [ ] `IngestSupportMessageInput` is shared between bot, workflow, and activity with no type drift
