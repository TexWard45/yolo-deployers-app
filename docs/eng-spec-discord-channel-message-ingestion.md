# Engineering Spec: Discord Channel Message Auto-Ingestion

## 1. Job to Be Done

- **Who**: Workspace admins who want to capture Discord channel messages as support conversations automatically.
- **What**: The system should listen to configured Discord channels via a bot, receive every non-bot message in real time, and persist it into the database as a `ConversationMessage` linked to a `Conversation` and `CustomerProfile` тАФ without any manual intervention.
- **Why**: The original Discord bot (`apps/queue/src/discord-bot.ts`) did inline ingestion directly in the `MessageCreate` handler with several problems:
  1. No Temporal workflow orchestration тАФ duplicated all ingestion logic inline.
  2. No auto-draft trigger тАФ `WorkspaceAgentConfig.autoDraftOnInbound` never fired.
  3. No retry/resilience тАФ DB write failures caused silent message loss.
  4. Channel filtering was hardcoded to channel names containing "support" or "company".
  5. No per-channel connection scoping via `ChannelConnection.configJson`.
- **Success criteria**:
  - Every non-bot message in a configured Discord channel is persisted as a `ConversationMessage` within seconds.
  - Messages are processed through the Temporal `ingestSupportMessageWorkflow`, enabling auto-draft generation.
  - Duplicate messages are never stored (idempotency via `externalMessageId`).
  - Channel filtering is driven by `ChannelConnection.configJson` (`channelIds`), not hardcoded name matching.
  - The system recovers gracefully from transient failures via Temporal retry policies.
  - On startup, the bot backfills recent messages from Discord channels, skipping already-saved ones.

## 2. Architecture

### Data Model

No new Prisma models. Existing models used:
- `ChannelConnection` тАФ `configJson` standardized to `{ channelIds: string[], listenToThreads?: boolean }`
- `Conversation`, `ConversationMessage`, `CustomerProfile`, `CustomerChannelIdentity` тАФ unchanged

                    +-----------------+

### Startup Sequence

1. Worker starts Temporal worker + Discord bot
2. Bot logs in, fires `ClientReady`
3. Auto-discover: for connections with old/missing `configJson`, fetch all text channels from the Discord guild via API and update the DB record
4. Refresh connection cache (in-memory, 60s TTL)
5. Start one long-running `ingestSupportMessageWorkflow` per active connection
6. Backfill: fetch last 100 messages per channel from Discord API, skip already-saved (idempotency check via `externalMessageId` in DB), signal remaining to the workflow
7. Begin real-time listening via `MessageCreate` events

## 5. Key Design Decisions

### Why `signalWithStart` instead of `workflow.start` + `signal`?

Calling `start` and `signal` separately creates a race condition when multiple messages arrive concurrently (e.g., during backfill). Each call to `ensureWorkflowRunning` could see no running workflow and start a new one, resulting in dozens of duplicate workflows. `signalWithStart` is atomic: it starts the workflow if it doesn't exist, or signals the existing one.

### Why a signal-driven infinite workflow instead of one workflow per message?

- One workflow per message creates thousands of short-lived workflows, each with overhead.
- A single long-running workflow per connection batches processing naturally.
- `continueAsNew` at 500 messages prevents Temporal history from growing unbounded.
- The workflow maintains an internal queue, processing messages sequentially тАФ natural backpressure.

### Why auto-discover channels on startup?

Existing `ChannelConnection` records had old-format `configJson` (e.g., `{"channelFilter": "..."}`) without `channelIds`. Rather than requiring manual migration, the bot auto-discovers text channels from the Discord guild via API and updates the DB record. This is a one-time self-healing migration.

### Why backfill on startup?

If the bot was offline, messages were missed. On startup, it fetches the last 100 messages per channel from Discord's API, checks each against the DB for idempotency (via `externalMessageId`), and signals any new ones to the workflow. This ensures no gaps in conversation history.

### Why an explicit `pg.Pool` in `@shared/database`?

The `PrismaPg` adapter factory (Prisma 7) has two modes:

1. **Config mode** тАФ `new PrismaPg({ connectionString })` тАФ creates a **new `pg.Pool` on every internal `.connect()` call**, then tears it down after each session. Under Temporal activity workloads with many sequential operations, the rapid pool create/teardown cycle caused intermittent SASL authentication failures (`client password must be a string`) because the ephemeral pool's connection handshake would occasionally fail.

2. **External pool mode** тАФ `new PrismaPg(pool)` тАФ reuses the provided `pg.Pool` across all `.connect()` calls. The pool manages its own connection lifecycle with proper keep-alive and reuse.

The fix: create a single `pg.Pool` in `@shared/database` and pass it to `PrismaPg`. This eliminated all SASL errors and is the recommended pattern per Prisma 7 docs for long-running services.

```ts
const pool = new pg.Pool({ connectionString: url });
const adapter = new PrismaPg(pool);
return new PrismaClient({ adapter });
```

### Why seed the Discord connection on queue worker startup?

The queue worker (and its Discord bot) depend on a `ChannelConnection` record existing in the database. After a fresh database (e.g., Docker container recreated), the worker would start with 0 active connections and silently do nothing.

`apps/queue/src/seed-connection.ts` runs at the beginning of `runWorker()` and idempotently ensures:
- A "yolo-deployers" `Workspace` exists
- An admin `User` + `WorkspaceMember` (OWNER) exist
- A Discord `ChannelConnection` (guild `1478163776506171588`, status `active`) exists
- A `WorkspaceAgentConfig` (with `autoDraftOnInbound: true`) exists

All upserts use find-first-then-create, so repeated startups are safe. The bot's auto-discover then updates `configJson.channelIds` from the Discord API on first run.

## 6. Testing Checklist

### Happy Path

- [ ] Send a message in a configured Discord channel тАФ verify `ConversationMessage` created
- [ ] Send from a new Discord user тАФ verify `CustomerProfile` + `CustomerChannelIdentity` created
- [ ] Send from a returning user тАФ verify existing profile reused
- [ ] Send when `autoDraftOnInbound` enabled тАФ verify `ReplyDraft` created

- [ ] Duplicate message тАФ idempotency prevents duplicate `ConversationMessage`

- [ ] Inactive `ChannelConnection` тАФ messages ignored
- [ ] Temporal unavailable тАФ bot logs error, does not crash
- [ ] Connection cache refresh тАФ new/removed connections take effect within 60s
- [ ] `continueAsNew` fires after 500 messages тАФ workflow continues seamlessly
